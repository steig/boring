How it worksFrom git clone to a working dev loop in six steps.

Here’s the actual walkthrough from the v0.1 dogfood against a production Shopify theme — no fictional examples.

  1. Install boring + dependencies (one-time per machine). brew install devcontainer, install Orbstack or Docker Desktop, install boring (curl install when the repo is public; clone-and-PATH for now). boring doctor reports any gaps.
  2. Drop a profile into the repo. .boring/profile.yaml declares theme: shopify, the mounts to share host CLI auth, ports to forward, and any per-project env. About 30 lines. Reviewable in a PR.
  3. Open it. boring open . parses the profile, generates .devcontainer/docker-compose.yml + devcontainer.json, and brings up a container with Node 20, Ruby 3.3, Shopify CLI, gh, and Claude Code preinstalled. First run ~60s; reopens ~5s.
  4. Authenticate, once. Inside the container: shopify auth login uses the device-code flow — the CLI prints a URL, you open it in your host browser, approve. Tokens persist to your host via the bind-mount.
  5. Develop with hot-reload + Claude in the loop. npm run dev serves the dev store at localhost:9292 in your host browser. Edit Liquid/CSS/JS in VS Code (attached via Dev Containers extension); Claude inside the container is project-scoped — ask it to make a change, see it in the preview.
  6. Ship. Commit on a feature branch inside the container, gh pr create. The PR lands in your repo, engineers review, your team’s normal deploy workflow takes over.

Headless agent runs (boring run <profile> --task …) and `--learn-mode` egress allowlists are on the roadmap, not in v0.1.

Why boringThe boring choices, finally.

Most "make dev easier" tools cheat by skipping the hard parts — isolation, real data, secrets, network safety — and you find out at the worst possible moment. boring is built around three priorities, in this order:

Security first Per-project egress allowlists block prompt-injection exfiltration before it happens. Sensitive data restores are ephemeral and never sit on disk between sessions. Secrets live in your OS keyring (or 1Password, or Vault — pluggable). Audit log of every sensitive restore in case a laptop walks off.
Practicality second One CLI. Stateless. Compose sidecars under the hood (boring-supported, not boring-customized). The profile lives in your repo as .boring/profile.yaml — reviewable in a PR, versioned with your code, the single source of truth. No drift across teammates' laptops.
Speed third First open in ~90 seconds. Subsequent opens in ~5 seconds. Container stays warm. Source code lives in a bind-mount so host-side and container-side git are the same git. Orbstack on Mac means near-native file performance.

Anatomy of a profile.boring/profile.yaml

Everything boring needs to recreate the dev environment lives in one file. Below is the actual v0.1 profile shipping for a production Shopify theme — this is what runs, not a marketing mockup. 

# .boring/profile.yaml
name: my-shopify-theme
theme: shopify           # picks the Shopify Dockerfile preset + egress defaults

services: []              # Shopify hosts the data; no sidecars in v0.1

# Share host CLI auth into the container. RW so `shopify auth login`
# and `gh` can persist tokens back to the host.
mounts:
  - ~/.config/shopify:/home/dev/.config/shopify
  - ~/.config/gh:/home/dev/.config/gh

# 9292 = `shopify theme dev` hot-reload; 3456 = OAuth callback (if used)
forward_ports: [9292, 3456]

env:
  SHOPIFY_STORE: your-dev-store
  SHOPIFY_FLAG_STORE: your-dev-store    # both names: npm scripts + CLI native

# Per ARD-0005, in-container guardrails refuse pushes that would touch prod.
# Codegen of pre-push hooks + command wrappers ships in the next slice;
# the schema is parsed today.
guardrails:
  forbid_branches: [main, dev-preview]
  forbid_commands:
    - "gh pr merge"
    - "shopify theme push --live"

data_sensitivity: internal   # no DB to gate in v0.1; matters for Django (v1.x)

FeaturesWhat ships today, what’s next.

Honest tags. today = working in v0.1, validated end-to-end. roadmap = designed, scheduled, not yet shipping.

today
Profile-in-repo

.boring/profile.yaml is the single source of truth. Sharing the profile is sharing the repo — no export/import dance, no drift across laptops. Schema validated, overlay merge for user-local overrides.

today
Theme presets

theme: shopify selects a curated Dockerfile (Node 20, Ruby 3.3, Shopify CLI, gh, Claude Code) with non-root user, port-forwards, and the egress-allowlist defaults that match the tool. Django, Rails, and others land next.

today
Project-scoped AI

Claude lives inside the container, with project-scoped MCP servers, memory, and conversation history. A poisoned file in one project can’t read another project’s notes. API key resolves from your existing secret store.

today
Pluggable secrets resolver

boring owns zero secret storage. The profile declares URIs into your store: op://, keychain:, vault://, aws-sm:, dbx-vault:, env:, file:. Resolved at container start. Nothing in .env. Nothing in shell history.

today
boring doctor

One command diagnoses runtime versions, devcontainer CLI, dbx, and the optional URI-resolver tools (op, vault, aws, security, secret-tool). Tech leads stop being the dev-environment helpdesk.

today
VS Code & Codespaces compatible

Because boring generates a standard devcontainer.json, your team can open the project with VS Code’s Dev Containers extension — or in GitHub Codespaces — with no boring CLI installed on their end.

roadmap
Egress allowlist enforcement

Schema is parsed today; iptables-in-container vs. proxy-sidecar prototype lands in v1.x. --learn-mode will observe a session and propose the right allowlist based on what your app actually called.

roadmap
Guardrails codegen

The guardrails: block (forbidden branches, forbidden commands, allowed Claude tools) is parsed today. Codegen of in-container pre-push hooks + command wrappers ships next — so accidental pushes to main are mechanically impossible.

roadmap
Sidecars + dbx restore

Compose sidecars (Postgres, Redis, etc.) with dbx restore piping prod-shape data into a named container. Ephemeral DB volumes auto-derived from data_sensitivity. The Django use case.

roadmap
Headless boring run

The same container shape, invoked as a one-shot autonomous Claude task from CI, a Slack bot, or your terminal. Shares the egress allowlist, vault, and guardrails with the interactive entry point.

RoadmapWhere it goes from here.

v0.1

Shopify themes

Shipping today. boring open end-to-end against a production Shopify theme: profile parser, compose generator, Shopify Dockerfile preset, secret URI resolver, boring doctor. Plumbing validated; non-engineer + Claude can iterate the theme and submit PRs.

v1.0

Django + sidecars

The case boring was originally designed for. Compose sidecars (Postgres, Redis), dbx restore --into <sidecar>, env-var rewrites so the wrapped app talks to sidecar hostnames without editing .env, ephemeral DB volumes derived from data_sensitivity.

v1.x

Guardrails + egress

In-container codegen of pre-push hooks + command wrappers from guardrails:. Egress allowlist enforcement (iptables or proxy sidecar). --learn-mode proposes the right allowlist by observing one session. Headless boring run.

v1.x

Distribution & speed

Public curl | bash install. Theme preset images published to ghcr.io so first-run drops from ~60s to ~5s. Auto-recreate the container when the compose file changes. brew + winget formulas.

v2

Team telemetry

Optional opt-in metrics: first-open duration, build times, failure stages. So you can actually measure whether the dev experience is improving instead of guessing.

Get startedv0.1 works today.

The repo is private during v0 development — access is by invite while we shake out the edges. Once public, install will be one line; for now it’s a clone. 

# prerequisites: docker (Orbstack on Mac), then:
$ brew install devcontainer
$ git clone git@github.com:steig/boring.git ~/code/boring
$ export PATH="$HOME/code/boring:$PATH"
$ boring doctor

# in any boring-enabled repo (one with .boring/profile.yaml):
$ boring open .
